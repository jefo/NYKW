# NYKW Backlog

## Эпик: /audit — Проверка необоснованных решений

**Статус:** будущий (не в MVP)

### Контракт: стек проекта = accepted ADR

Каждая зависимость проекта (npm-пакет, база данных, фреймворк) — это ADR со статусом `accepted`.
ADR со статусом `accepted`, у которых отсутствует обоснование в секции "# Решение" или тело пустое, считаются **необоснованными** и подлежат аудиту.

Команда `/audit`:
1. Сканирует все ADR со статусом `accepted` через `nykw query --status accepted --format json`
2. Анализирует тело ADR на наличие обоснования в секциях "# Решение" и "# Контекст"
3. Выявляет ADR с пустыми или шаблонными обоснованиями
4. Запускает интерактивный воркфлоу с агентом: помогает выбрать технологию, задаёт вопросы, предлагает альтернативы
5. Итог: либо ADR дополняется обоснованием, либо статус меняется на `deprecated`

### Фича: Cross-Layer Impact Analysis

Если ADR затрагивает слой `infra`, то какое влияние это оказывает на слой `application` и слой `domain`?

Примеры коллизий:
- ADR "Используем PostgreSQL" (infra) → Application не должен писать сырые SQL-запросы в контроллерах
- ADR "Кэшируем профили через Redis" (infra, application) → Domain не должен знать о Redis
- ADR "Внедряем Event Sourcing" (domain, infra) → Application слой должен перейти на асинхронные проекции

Агент `/audit` должен:
1. При анализе ADR со слоем `infra` — проверять, есть ли связанные ADR в `application` и `domain`
2. Предлагать создать дочерние ADR для смежных слоёв
3. Строить матрицу `layer × layer` для визуализации связей

### Фича: Emergent Architectural Patterns

ADR создаются не только для технологического стека, но и когда проект развивается функционально. Мы отслеживаем архитектурные паттерны и практики, которые **впервые** применяются в проекте.

Что попадает под это:
- Первый `Repository<T>` → ADR о выборе паттерна Repository
- Первый `Facade` для внешнего сервиса → ADR о паттерне Facade для интеграций
- Первый случай CQRS → ADR о разделении команд и запросов
- Первый Event-Driven хендлер → ADR о переходе на события
- Первый Feature Flag → ADR о Feature Toggle стратегии
- Первый Unit of Work → ADR о транзакционных границах

Отличие от технологических ADR:
- Технологический ADR: "Выбрали PostgreSQL" (infra) — отвечает на *что*
- Паттерн-ADR: "Применили Repository Pattern" (domain/application) — отвечает на *как*
- Практика-ADR: "Все HTTP-вызовы через Retry Policy" (application) — отвечает на *как*

Агент:
1. Сканирует кодовую базу на известные паттерны (сигнатуры классов, декораторы, именования)
2. Сверяет с ADR-графом: есть ли ADR, объясняющий *почему* паттерн используется
3. Если паттерн впервые и нет ADR → предлагает создать

### Фича: /scan — обнаружение недокументированных паттернов

Более лёгкая версия `/audit`, без интерактивного воркфлоу. Просто сканирует код и сообщает: «Вижу 3 недокументированных паттерна: Repository, Facade, CQRS. Запусти `/audit` для разбора.»
